# Product ingestion strategy

Цель: получать товары, цены и наличие из внешних источников так, чтобы проект был устойчивым, масштабируемым и юридически безопасным.

Базовый продуктовый алгоритм находится в [`universal_tech_price_parser_algorithm.md`](../universal_tech_price_parser_algorithm.md). Этот документ описывает production-адаптацию: источники, ограничения, очереди, хранение и правила включения коннекторов.

## Принцип

Мы не обходим `robots.txt`, CAPTCHA, антибот-защиту, rate limits и закрытые API. Если источник запрещает автоматический сбор или начинает отвечать лимитами, коннектор останавливается и требует ручного решения: партнерский фид, официальный API, договоренность с магазином или отключение источника.

## Приоритет источников

1. Официальный API магазина или маркетплейса.
2. Партнерский товарный фид.
3. Merchant/YML/XML/CSV feed от магазина.
4. Sitemap и публичные страницы, если это разрешено `robots.txt`.
5. Ручная загрузка прайсов через админку.

HTML-парсинг публичных страниц допустим только как fallback и только после проверки `robots.txt`.

## Архитектура пайплайна

```text
Source registry
  -> robots/policy guard
  -> discovery
  -> fetch queue
  -> parser adapter
  -> normalizer
  -> product matcher
  -> quality scoring
  -> moderation queue
  -> database upsert
  -> price history
```

## Source registry

Для каждого источника храним:

- `slug`;
- `baseUrl`;
- тип источника: `api`, `feed`, `sitemap`, `html`, `manual`;
- разрешенные категории;
- лимит запросов;
- минимальный интервал между запросами;
- правила парсинга;
- контакт/договор/партнерский аккаунт, если есть.

## Discovery

Discovery отвечает за список URL или offer records.

Варианты:

- API pagination;
- XML/YML feed;
- sitemap;
- заранее заданные category URLs;
- ручная загрузка CSV/XLSX.

Discovery не должен ходить глубже, если `robots.txt` запрещает путь.

## Fetch queue

Очередь должна быть на BullMQ/Redis.

Обязательные правила:

- один домен не обрабатывается параллельно без отдельного разрешения;
- есть per-domain throttle;
- есть backoff на `429`, `403`, `503`;
- после нескольких ошибок источник переводится в статус `paused`;
- ответы кешируются для повторного парсинга без повторного запроса.

## Parser adapter

Каждый источник получает отдельный адаптер:

```text
DnsAdapter
CitilinkAdapter
MvideoAdapter
AvitoAdapter
YandexMarketFeedAdapter
ManualFeedAdapter
```

Адаптер возвращает не финальный товар, а сырой `RawOffer`.

```ts
type RawOffer = {
  source: string;
  externalId?: string;
  url: string;
  title: string;
  price?: number;
  currency: "RUB";
  imageUrls: string[];
  categoryHint?: string;
  brandHint?: string;
  sellerName?: string;
  condition?: "new" | "used" | "unknown";
  rawSpecs: Record<string, string>;
  collectedAt: string;
};
```

## Normalization

Нормализация приводит данные к единому виду:

- чистит название;
- выделяет бренд;
- определяет модель;
- приводит характеристики к словарю категории;
- нормализует цену;
- удаляет маркетинговый шум;
- рассчитывает fingerprint товара.

Пример fingerprint:

```text
category + brand + normalized_model + key_specs
```

Для процессоров:

```text
processors + amd + ryzen-9-7950x + socket-am5
```

## Product matching

Матчинг должен быть осторожным.

Сигналы:

- точный артикул/SKU;
- EAN/GTIN, если есть;
- нормализованная модель;
- бренд;
- ключевые характеристики;
- похожесть названия;
- категория.

Если уверенность ниже порога, запись идет в moderation queue, а не сливается с существующим товаром.

Рекомендуемые пороги:

- `>= 0.92` - авто-матч;
- `0.75-0.92` - на модерацию;
- `< 0.75` - новый кандидат или discard.

## Avito quality scoring

Авито нельзя смешивать с магазинами без фильтра качества: там частные продавцы, б/у товары, копии, битые карточки и странные описания.

Для Avito делаем отдельный scoring.

Положительные сигналы:

- есть фото;
- цена в разумном диапазоне категории;
- название содержит бренд и модель;
- описание не пустое;
- продавец с историей/рейтингом, если доступно легальным способом;
- объявление не старое;
- есть доставка или безопасная сделка, если доступно.

Отрицательные сигналы:

- цена ниже медианы категории больше чем на 50%;
- в названии есть слова `копия`, `реплика`, `муляж`, `не включается`, `на запчасти`, `битый`;
- нет фото;
- слишком короткое описание;
- слишком много капса, эмодзи, телефонов, внешних контактов;
- категория не совпадает с моделью;
- seller/location резко не совпадает с условиями источника;
- дублируется по фото/названию много раз.

Решение:

- `score >= 80` - можно показывать как offer с пометкой `частное объявление`;
- `50-79` - на модерацию;
- `< 50` - не импортировать.

## Защита от мусора

Минимальный набор:

- дедупликация по URL и external ID;
- дедупликация по title fingerprint;
- проверка ценовых выбросов;
- проверка обязательных характеристик категории;
- quarantine table для подозрительных offer;
- ручная модерация до попадания в публичный каталог.

## Первые источники

### DNS

Начинать с проверки sitemap и разрешенных путей. Если каталог разрешен, брать только ограниченный набор категорий и очень низкую частоту. Лучше запросить партнерский/товарный фид.

### Citilink

Есть партнерские программы и виджеты, поэтому основной путь - партнерский фид/договор. Если сайт отвечает `429` даже на проверочные запросы, источник ставится на паузу.

### М.Видео

Основной путь - официальный/партнерский фид или маркетплейс-интеграция. HTML fallback только после проверки `robots.txt`.

### Avito

Основной путь - официальный API/партнерский доступ. Без этого Avito должен идти через ручной импорт или отключаться. Даже при легальном доступе все объявления проходят quality scoring.

## MVP ingestion

1. Таблица источников.
2. Robots guard.
3. Feed/manual adapter.
4. Нормализованная целевая карточка товара.
5. Один официальный/ручной источник.
6. Normalizer для текста, цены, емкости и наличия.
7. Category-specific matcher для CPU/GPU/SSD/RAM.
8. Confidence scoring.
9. Price outlier detection.
10. Market price report: median, average, p25/p75, min trusted.
11. Price history.
12. Moderation queue.

HTML-коннекторы к крупным сайтам добавлять только после легальной проверки источника и понятного rate limit.
