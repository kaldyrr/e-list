# Инструкция по работе команды

## Git-процесс

Основные ветки:

- `main` - продакшен-ветка. Прямые коммиты запрещены.
- `develop` - общая ветка разработки. Все фичи вливаются сюда через PR.

Стартовые рабочие ветки:

- `feature/backend-core` - backend, база данных, API, Prisma, авторизация.
- `feature/frontend-shell` - frontend shell, layout, дизайн-система, routing.
- `feature/catalog` - категории, товары, фильтры, сортировка.
- `feature/product-page` - карточка товара, характеристики, предложения магазинов.
- `feature/compare-favorites` - сравнение товаров и избранное.
- `qa/e2e-smoke` - тест-план, smoke/e2e, регрессия.

Для каждой новой задачи создается отдельная ветка от `develop`:

```bash
git checkout develop
git pull
git checkout -b feature/short-task-name
```

Используйте префиксы:

- `feature/` - новая функциональность;
- `fix/` - исправление бага;
- `chore/` - техническая задача без изменения поведения;
- `docs/` - документация;
- `qa/` - тесты, тест-планы, e2e.

## Pull request

Каждый PR открывается в `develop`.

Минимальный шаблон описания PR:

```md
## Что сделано

- ...

## Как проверить

- ...

## Скриншоты

Для UI-изменений обязательно.

## Риски

- ...
```

Правила:

- один PR решает одну задачу;
- PR должен быть небольшим и проверяемым;
- для UI-изменений прикладываются скриншоты desktop и mobile;
- для backend-изменений описываются новые/измененные endpoints;
- если меняется схема БД, в PR указывается миграция и влияние на существующие данные;
- минимум один review перед merge;
- merge только после успешных проверок.

## Commit-сообщения

Используем короткий conventional-style формат:

```text
feat: add product listing
fix: correct price sorting
chore: configure eslint
docs: update workflow
test: add product page smoke test
```

## Разделение задач

### Backend developer

Отвечает за:

- PostgreSQL schema;
- Prisma models и migrations;
- NestJS modules;
- API contracts;
- импорт предложений магазинов;
- поиск, фильтрацию, сортировку;
- авторизацию и роли;
- backend-тесты.

### Fullstack 1

Отвечает за:

- категории;
- каталог;
- фильтры;
- сортировку;
- интеграцию frontend с backend API;
- состояния loading/empty/error.

### Fullstack 2

Отвечает за:

- карточку товара;
- характеристики;
- сравнение товаров;
- избранное;
- историю цен;
- интеграцию с предложениями магазинов.

### Frontend 1

Отвечает за:

- Next.js app shell;
- layout;
- навигацию;
- дизайн-систему;
- общие компоненты;
- адаптивный каркас.

### Frontend 2

Отвечает за:

- товарные карточки;
- листинг;
- UI фильтров;
- страницы категорий;
- mobile states;
- visual polish.

### QA tester

Отвечает за:

- тест-план MVP;
- smoke-тесты;
- e2e Playwright-сценарии;
- регрессию перед релизом;
- проверку адаптива;
- баг-репорты в issue.

## Definition of Done

Задача считается готовой, если:

- код влит через PR;
- нет падающих тестов;
- нет TypeScript-ошибок;
- UI проверен на desktop и mobile, если задача фронтовая;
- API проверен через тест или ручной сценарий, если задача backend;
- добавлены или обновлены тесты там, где меняется поведение;
- документация обновлена, если поменялись команды, API или процесс.

## Ритм работы

Рекомендуемый цикл: недельные спринты.

Понедельник:

- планирование задач;
- назначение владельцев;
- фиксация цели недели.

В течение недели:

- короткий ежедневный статус;
- PR открывается как можно раньше;
- QA проверяет готовые куски, не ждет конец недели.

Пятница:

- демо работающего инкремента;
- регрессия;
- решение, что идет в релиз.

## Приоритеты MVP

1. Скелет монорепы и инфраструктура.
2. База данных: категории, товары, характеристики, магазины, предложения.
3. API каталога и карточки товара.
4. Главная и навигация.
5. Листинг с фильтрами.
6. Карточка товара.
7. Сравнение и избранное.
8. Админка.
9. E2E smoke.

## Запреты

- Не коммитить напрямую в `main`.
- Не вливать PR без review.
- Не делать большие PR "на все сразу".
- Не менять чужую рабочую область без согласования.
- Не добавлять новую библиотеку без причины и обсуждения.
- Не хранить секреты в репозитории.
